Case Study · 02

smriti

Long-memory companions for local LLMs. Conversations that span months, not minutes. Your data, your machine.

Repo on GitHub →

smriti (स्मृति) is Sanskrit for “memory.”

There is a category of conversation people want to have with an AI but not with a cloud provider. Therapy-adjacent. Health questions. Family matters. Things you would not type into ChatGPT because you have read the privacy policy.

The standard answer is local LLMs. Run Ollama on your Mac, point a chat client at it, no data leaves the machine. Beautiful in theory. In practice, local models on Apple Silicon have a small effective context window. Once a conversation passes eight thousand to thirty-two thousand tokens, the model starts forgetting. An hour of meaningful conversation, and the thread is gone.

smriti sits between you and a local LLM, archives older messages, embeds them, and pulls the relevant ones back into context when you need them. The conversation feels long. The data stays local. No accounts, no telemetry, no signup.

Design

Scopes

Most memory systems treat memory as one global pool. smriti uses scopes: user-defined memory partitions, each with its own system prompt. You might have a “Family Companion” scope, a “Research Notes” scope, a “Coding Helper” scope. Conversations within a scope share memory. Across scopes, never.

A general-purpose memory layer either remembers everything in one pile and contaminates every conversation with everything you have ever said, or remembers nothing and is no better than a long context window. Scopes are the middle path. You decide what context belongs together.

A scope is also a persona container. The system prompt for “Coding Helper” is different from “Family Companion.” The bot that helps you debug Rust is not the same bot that listens to you talk about your mother. It should not be.

Privacy

The privacy threat model

Privacy claims need to be precise. With the bundled chat UI plus Ollama, nothing leaves the machine. Messages, embeddings, summaries, all local. Postgres binds to localhost. Ollama binds to localhost. You can verify with lsof or Little Snitch.

With external MCP clients (Claude Desktop, Cline with cloud models), the smriti MCP server is still local, but those clients send your messages, and any memories retrieved on your behalf, to their respective cloud providers. This is fine if you trust them. It is not the local-only mode. The README says this in plain language. The distinction is the whole point.

Architecture

One service, two transports

The interesting design decision is that the same memory service is exposed through two surfaces. A FastAPI HTTP server feeds a React chat UI, the path with the full privacy guarantee. A FastMCP server speaks MCP over stdio for external clients like Claude Desktop. Same memory service underneath. Different transports.

Building two separate memory layers would have been the obvious move. The memory service is the core; the two surfaces are thin wrappers over it. One implementation, two transports. When the retrieval scoring changes, both surfaces get it. When the schema migrates, neither breaks.

Retrieval

Retrieval

Messages embed with nomic-embed-text (768 dimensions). Episodes (either individual messages or summaries of older message ranges) live in one Postgres table with pgvector HNSW indexes. Retrieval searches both kinds transparently.

Scoring is not cosine similarity alone. It is multi-component: similarity, recency, importance, reinforcement, and frequency. A message from yesterday about your current project ranks higher than a semantically similar message from six months ago about something you have not mentioned since. A message you returned to multiple times is probably more important than one you mentioned once. Cosine similarity alone misses all of this.

Status

Status

v1 is shipped and running. The repo is open source and the roadmap is public. Build it yourself if you want to run it. The whole point of a privacy-first tool is that you can read the code that handles your data.

Active development is ongoing. If you would like access to the work in progress or want to collaborate, email me.
