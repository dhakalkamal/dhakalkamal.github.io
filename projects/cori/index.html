Case Study · 01

CORI

Counterparty Risk Intelligence for community banks. A sanctions screening platform built end to end from public data.

Source code →

Community banks have the same regulatory obligations as JPMorgan when it comes to sanctions screening, and the same penalties for missing a hit. What they do not have is the same budget. Refinitiv World-Check and LexisNexis Bridger are priced for tier-one institutions at fifty to two hundred thousand dollars a year. A typical community bank’s entire compliance budget is smaller than that.

The data underneath these commercial products is mostly public. OpenSanctions republishes 179 upstream lists. GLEIF publishes 1.77 million LEI-issued legal entities. ICIJ Offshore Leaks is a download. GDELT is a 21 TB BigQuery public dataset. CORI is what it looks like to integrate those sources, score them, and present the result to a compliance analyst with the explanation an OCC examiner will demand.

Architecture

The architecture

A Bronze, Silver, Gold medallion pattern with PostgreSQL triggers enforcing append-only Bronze at the database level (audit integrity in the substrate, not in convention). Silver as Parquet with CSV companions. Gold not as a feature table but as a scoring output table, where a feature_version column ties every score to a reproducible Silver state and a top_factors JSONB column persists SHAP attributions alongside each prediction. Score and explanation travel together.

Storage splits across substrates by cost shape: PostgreSQL for Bronze audit and Gold scoring (under 5 GB total), BigQuery as external Bronze for GDELT (21 TB scanned, under twelve dollars in query cost), Parquet for Silver analytical tables.

Methodology

Why PU learning

Classical binary classification does not fit this domain. Sanctioned entities are confirmed positives. Absence of a listing does not confirm an entity is clean; it may not have been designated yet. Training a model that treats every unlabeled entity as negative teaches it that future sanctions targets are safe. Positive-Unlabeled learning, following Elkan and Noto, estimates the probability that an unlabeled entity is positive given that some true positives are hidden in the unlabeled pool. The positive class is 56,652 Tier 1 sanctions entities. The unlabeled class is 1.77 million GLEIF commercial entities. XGBoost, 500 trees, early stopping at 50.

Iteration

Four iterations and a methodology

The first model achieved an AUC of 1.0 on the source leakage gate. That sounded great until you understood what it meant. A single feature, tier_1_dataset_count, perfectly separated the populations. SHAP showed it commanding 100 percent of total attribution. The model had learned a lookup table.

Version 2 removed the obviously leaked features. The is_offshore feature took over at 90 percent SHAP share. Investigation revealed why: is_offshore was 100 percent null for OpenSanctions entities and 0 percent null for GLEIF. The feature was encoding source membership through its missingness, not offshore risk.

Removed is_offshore. entity_age_days took over at 90 percent. Same pattern. The two populations had systematically different age distributions and the model exploited that gap.

This was the moment that mattered. Removing leaked features one at a time was a losing strategy because the leakage was structural. When your positive and unlabeled populations come from different data sources, any feature can become a source proxy. The feature space itself had to be unified.

Version 3 ran ICIJ matching across both populations through the same code path. Version 3.1 added source-wise normalization, median imputation for nulls, and nationality fallback for missing jurisdiction codes. Source leakage AUC dropped from 1.0 to 0.81, above the 0.75 failure threshold but partially mitigated. Post-train dominance dropped to 42 percent. Twelve nonzero SHAP features across country risk, jurisdiction, offshore status, entity age, and ICIJ connections.

The leakage is documented, measured, partially mitigated, and transparently reported. That last part is the contribution.

Validation

Twelve gates

The iteration story is one specific leakage path. The broader lesson is that ad-hoc inspection does not scale. So we built a validation framework. Twelve gates, each testing one leakage or quality condition. A model must pass all twelve to be promoted to Gold. Pre-training gates cover lineage, single-feature determinism, univariate predictive power, source leakage AUC, and missingness gaps. Training-process gates cover coverage parity, family collinearity, split hygiene, and adversarial validation. Post-training gates cover dominance, calibration, and artifact integrity. The framework runs automatically before any model is promoted. Gate results are logged to the pipeline audit schema. An examiner asking how the model was validated gets a paper trail.

Network Analysis

The graph

A SHAP waterfall tells an analyst which features drove a risk score. For ICIJ-derived features it is not enough. A compliance officer needs to see what the connection is, to whom, and how many hops away a sanctioned party sits.

The ICIJ Offshore Leaks Database loads into PostgreSQL as a traversable graph: 2 million nodes, 3.3 million edges. For each scored entity with an ICIJ bridge match, breadth-first search finds the shortest path to any sanctioned node up to four hops. 6,551 entities have at least one exposure path. The interesting ones are the 784 entities at one to four hops, whose sanctions exposure is invisible without traversal.

The case that matters most: Sea Knot Shipping Inc. receives a score of 0.29 (LOW). The model considers it safe. The graph reveals a two-hop path to sanctioned entities via MOSSFON SUBSCRIBERS LTD, a Mossack Fonseca entity from the Panama Papers. An entity no score-only system would flag has hidden offshore connections worth investigating. The three-layer explanation (score, SHAP, graph) gives the analyst what they need to decide.

Results

What is running

The system scores 1,833,779 entities across three bands: 54,216 high, 2,649 medium, 1,776,914 low. Every score carries top-5 SHAP attribution and, where applicable, top exposure paths. The full pipeline (migrate, ingest, dbt_build, score, graph_build) runs inside a Docker container, triggered from the frontend, with each step recorded in pipeline audit tables.

The frontend is React with role-based access matching a real community bank’s compliance org: BSA Analyst, BSA Officer, Auditor. Every design choice (append-only Bronze, versioned feature contracts, pipeline audit tables, SHAP top-5 persistence, the twelve-gate framework) exists to answer the one question an OCC examiner can ask at any time: “Why this score?”
